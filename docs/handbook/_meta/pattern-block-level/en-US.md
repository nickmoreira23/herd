# Pattern: Block

Block is the data level of HERD ‚Äî single source of truth for a record type. Blocks live in families generated by tools (see `pattern-tool-level`); this pattern explains the internal structure of a family, naming conventions, and the canonical suffixes that encode semantics.

## Business

Data is a company asset. When blocks consolidate data types into single sources, the company gains auditability, cross-feature analysis, and incremental evolution without fragmentation. When data scatters into "ad-hoc tables per feature", the cost of any cross-feature question explodes.

The practical consequence: each data type has **one** owning block. Other tools may reference it via foreign keys, but never duplicate. Recognition references profiles via FK, but does not store a local copy of the profile.

## Product

Blocks appear to the user as **collections**. That is why the naming is always plural: `products`, `contacts`, `meetings`, `recognition-events`, `points-balances`. Each block represents "the list of things of this type".

**Records** are the concrete instances that show up on screen ‚Äî a specific product ("Moon Milk"), a contact ("Maria Silva"), a points event ("Jo√£o earned 30 pts from sale X").

**Block Group** is an optional layer of internal organization: it groups related records inside the same block. Beverages is a Block Group inside the `products` block. Block Groups have their own light metadata (name, order) but do not create a new data type ‚Äî they are curation.

The rule for Block Group: apply only where internal organization helps. A Products family with 5,000 SKUs needs it. A Locations family with 10 addresses doesn't.

## Architecture

### Internal hierarchy

```
Block Family (generated by a Tool)
‚îî‚îÄ‚îÄ Block (data category)
    ‚îî‚îÄ‚îÄ Block Group (optional grouping)
        ‚îî‚îÄ‚îÄ Record (concrete instance)
```

Each level has a distinct responsibility:

- **Block Family**: set of correlated blocks under the same tool. Ex: Products family = `products` + `products-categories` + others as the tool grows.
- **Block**: a single conceptual table ‚Äî its own Prisma model, its own CRUD, independent lifecycle.
- **Block Group**: declarative intra-block grouping. No CRUD of its own. Lives in metadata of the owning block.
- **Record**: individual row in the block. What the user creates, edits, reads.

### Canonical suffixes

Suffixes in the block id encode semantics. Apply them whenever the semantics match:

| Suffix | Function |
|---|---|
| (no suffix) | Main block of the family (e.g., `products`, `contacts`) |
| `-categories` | Block Group naming |
| `-tracks` / `-systems` / `-rules` | Main definitions (Recognition, Ranking, Points) |
| `-levels` | Named levels (Recognition levels: Bronze, Silver, Gold) |
| `-criteria` | Reusable conditions (requirements to level up, eligibility rules) |
| `-events` | Time log (with source attribution) |
| `-progress` | Continuous current state (current level + profile history on a track) |
| `-positions` | Ranking current + frozen history per period |
| `-balances` | Dynamic current balance (updates on event) |
| `-snapshots` | Frozen history per configurable period |
| `-benefits` | Conditional rules active at each level |
| `-rewards` | One-off awards granted |
| `-stories` | Publishable narrative content |
| `-payouts` | Audited consolidated payments |
| `-redemptions` | Redemptions |
| `-transitions` | Log of state changes between related instances |

Suffixes are not merely cosmetic: they distinguish semantics that would otherwise be ambiguous. `recognition-events` is a log; `recognition-progress` is current state; `recognition-snapshots` is a historical photo. Three distinct blocks, three distinct responsibilities.

### Source attribution on events

Every block with the `-events` suffix carries a polymorphic `source` field pointing to the originating record. Full detail in `pattern-source-attribution` (Batch 3). Summary:

```
event {
  id, profile_id, value, occurred_at,
  source: { source_block, source_id, source_type }
}
```

Enables drill-down ("why did I earn 30 pts?" ‚Üí click ‚Üí opens the original sale), audit, and reversal when the source event is canceled.

### Distinction between state-block types

Common vocabulary conflates "current state" with "history". HERD distinguishes:

- `-progress`: continuous state, no period. Ex: current level in `recognition-progress`.
- `-balances`: dynamic balance, updates on event. Ex: `points-balances`, `remuneration-balances`.
- `-positions`: ranking current + frozen history per period. Specific to Ranking.
- `-snapshots`: frozen photo per configurable period. Generalizes to any state that needs periodic history.

Individual events always remain in `-events` as the source of truth ‚Äî the other blocks are derived projections.

### Profile vs profile-types vs profiles ‚Äî semantic distinction

A special case that shows how naming disambiguates:

| Term | Type | Location | Singular/Plural |
|---|---|---|---|
| Profile (institutional) | Standalone tool | Organization area | Singular |
| Profile (personal ‚Äî future) | Standalone tool | Identity area | Singular |
| `profile-types` | Block (managed via Network "Manage types") | Network tool | Plural |
| `profiles` | Block (instances) | Network tool consumes | Plural |

No collision ‚Äî different contexts (area, layer, singular vs plural) clarify. The same principle applies to any family that has "type + instance" (plan-types/plans, permission-sets/permissions).

## Operations

### Checklist for a new block

1. **Owning tool**: which tool does this block belong to? If none ‚Üí first define the tool (see `pattern-tool-level`); blocks do not live outside a family.
2. **Appropriate suffix**: is the block a time log? `-events`. Continuous current state? `-progress` or `-balances`. Historical photo? `-snapshots`. Rule definition? `-rules` or `-tracks`. No suffix if it is the main block of the family.
3. **Always-plural naming**: the block id is plural (`products`, not `product`). Reflects the fact that the block represents a collection.
4. **Block Group**: applicable? If yes, declare it inside the owning tool's `feature.yml`, in the `block_groups` field. Not a separate entry.
5. **Source attribution**: is the block an `-events`? Include a polymorphic `source` field. Without it, drill-down and reversal are impossible.
6. **Cross-references via FK**: what does this block reference (profiles, products, etc.)? Use a FK to the owning block ‚Äî never duplicate data.

### Anti-patterns to avoid

- **Singular block** (`product` instead of `products`). Always plural ‚Äî a block represents a collection.
- **Block outside a family**: creating an isolated block with no owning tool. Every data category lives in a family.
- **Missing suffix when applicable**: creating `recognition-history` instead of `recognition-snapshots`. Canonical suffixes have precise semantics; ad-hoc names fragment.
- **Block duplicating another block's data**: creating a local copy of profile inside Recognition instead of a FK to `profiles`. Duplication breaks the single source of truth principle.
- **Block-group as a separate entry**: creating a `_meta` or its own directory for a block-group. Block Groups are declarative metadata of the owning block.

## Glossary

- **Block**: data level ‚Äî single source of truth for a record type. Always plural.
- **Block Family**: set of correlated blocks generated/manipulated by a tool.
- **Block Group**: optional intra-block grouping; declarative metadata, no CRUD of its own.
- **Record**: concrete instance inside a block ‚Äî what the user creates/edits/reads.
- **Canonical suffix**: tag in the block id that encodes semantics (`-events`, `-snapshots`, `-progress`, etc.).
- **Source attribution**: polymorphic field on events pointing to the originating record (source_block + source_id + source_type).
- **Single source of truth**: principle that each data type has exactly one owning block; others reference it via FK.
