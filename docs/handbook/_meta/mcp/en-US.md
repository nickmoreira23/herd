# MCP Server

Model Context Protocol server that exposes the Handbook as a navigable resource for AI assistants. Supports two transports: **stdio** (for local clients like Claude Desktop and Claude Code) and **HTTP** (for remote clients like ChatGPT web). Both share the same tool logic (search + fetch) over the same `mcp/generated/search-index.json`.

## Business

The MCP server exists so AI assistants can answer questions about ComeçaAI using canonical documentation instead of guessing or inferring from code. When an assistant is asked "how does the ledger work in ComeçaAI?", it calls `search` to discover the entry, then `fetch` to read the full markdown — landing on the same content a human sees at `/admin/handbook/...`.

Without the MCP server, every new conversation starts from scratch: the assistant reads code, infers semantics, and produces plausible but non-canonical answers. With MCP, the assistant hits a versioned source (Handbook commits) and quotes what it read. This is particularly important for automated code review, spec generation, and onboarding new agents.

## Architecture

### Tools

Two tools registered:

- **`search(query)`** — lexical search over the index. Returns up to 20 results ranked by score (uid=10, id=8, title=5, description=3, body=1) with `{id, title, text, url}`.
- **`fetch(id)`** — returns full entry content by UID, with title/body/url/metadata in both locales.

The scoring logic is shared with the in-app client-side search via `src/lib/handbook/search.ts` — same pure function. Changes on one side reflect in both.

### Transports

#### Stdio (local clients)

Entrypoint: `mcp/index.ts`. Runs as a Node process via `npm run mcp:dev` (or whatever command is configured by the client). The client spawns the process, sends JSON-RPC over stdin, receives responses on stdout. Logs go to stderr.

Typical Claude Desktop config:

```json
{
  "mcpServers": {
    "comecaai-docs": {
      "command": "tsx",
      "args": ["/path/to/comecaai/mcp/index.ts"]
    }
  }
}
```

#### HTTP (remote clients)

Endpoint: `POST /api/mcp` (Next.js route handler in `src/app/api/mcp/route.ts`). The client sends JSON-RPC in the body; the server responds with Server-Sent Events compatible with the MCP Streamable HTTP spec.

Auth: static bearer token in the env `MCP_HTTP_TOKEN`. Without a token configured, the route returns `503`. Wrong token → `401`. Correct token → request is processed via the SDK's `WebStandardStreamableHTTPServerTransport`.

Mode: stateless (`sessionIdGenerator: undefined`). Each request is independent; no cross-request state. Acceptable for the current case (every `tools/call` is self-contained).

CORS: configured for `https://chatgpt.com` (the ChatGPT web origin). Other origins need explicit adjustment.

### Index generation

`mcp/generated/search-index.json` is generated by `scripts/build-search-index.ts` (part of `npm run gen:all`). It contains every entry with full bilingual bodies. The tools cache it in memory at module load. After a `git pull` that changes entries, the dev server needs a restart to invalidate the cache.

### Connection between transports

Both transports invoke the same `createServer()` function in `mcp/server.ts`, which registers the two tools. The only difference is the connected transport: `StdioServerTransport` vs `WebStandardStreamableHTTPServerTransport`. The pure `searchEntries` function in `src/lib/handbook/search.ts` is shared with the in-app client-side search via `/admin/api/handbook/search`.

## Operations

### Local setup — stdio

No additional config required. `npm run mcp:dev` brings up the stdio entrypoint. A local client (Claude Desktop, Claude Code) configured with `command: "tsx"` + absolute path to `mcp/index.ts` connects directly.

### Local setup — HTTP

1. Generate a token: `openssl rand -hex 32`.
2. Set the env var `MCP_HTTP_TOKEN=<generated-token>` (in `.env.local` or via direct export).
3. `npm run dev` (Next.js dev). Endpoint available at `http://localhost:3000/api/mcp`.
4. Smoke test:
   ```bash
   curl -X POST http://localhost:3000/api/mcp \
     -H "Authorization: Bearer $MCP_HTTP_TOKEN" \
     -H "Content-Type: application/json" \
     -H "Accept: application/json, text/event-stream" \
     -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}'
   ```

If `MCP_HTTP_TOKEN` is not set, the endpoint returns `503` ("MCP HTTP not configured") — stdio still works independently.

### HTTP deployment

The `/api/mcp` route deploys with the rest of the admin (same Next.js infra). `MCP_HTTP_TOKEN` must be configured in the production environment vars. CORS currently allows only `https://chatgpt.com` — other origins require editing `src/app/api/mcp/route.ts`.

### Debugging

- **Stdio**: stderr logs appear in the client's terminal.
- **HTTP**: logs go to the Next.js dev console. JSON-RPC errors come back in the response body (`{result:{isError:true, content:[...]}}`).
- **Index cache**: tools cache `search-index.json` in memory (module-level). After `gen:all` that changes the index, restart the dev server to invalidate.

### Tests

Automated coverage lives in `mcp/__tests__/` (vitest, runs with `npm test`). Integration-style: hits the real `mcp/generated/search-index.json` — same input the production server consumes. Avoids fixture↔prod drift.

- `search.test.ts` (11 cases): empty/whitespace queries, exact match, ordering by field weight, limit, snippet with `<mark>`, hierarchical URLs (layer/category/feature, layer-only, meta), bilingual.
- `fetch.test.ts` (6 cases): valid UID, unknown UID → error, hierarchical URL, layer URL, meta URL, bilingual metadata.
- `server.test.ts` (2 cases): smoke for `createServer()`.

**Not covered day-1** (deferred):
- HTTP route handler (requires a running Next.js server during the test).
- End-to-end stdio transport (requires spawning a child process).

Both layers are exercised via manual curl (HTTP) or by a real client connecting (stdio). Promote to automated tests when it starts to bite.

## Glossary

- **Bearer token**: Auth scheme where the client sends `Authorization: Bearer <token>` in the header. Day-1 of the HTTP transport uses a static token configured via env var.
- **JSON-RPC**: JSON-based RPC protocol used by MCP. Requests carry `{jsonrpc, id, method, params}`; responses carry `{jsonrpc, id, result}` or `{jsonrpc, id, error}`.
- **MCP**: Model Context Protocol — open spec defining how AI assistants consult external resources (tools, resources, prompts) in a uniform way.
- **Stateless mode**: Mode of the Streamable HTTP transport where no session ID is maintained between requests; every request is fully independent.
- **Stdio transport**: MCP transport where the server is a Node process spawned by the client; communication over stdin/stdout.
- **Streamable HTTP**: HTTP-based MCP transport supporting both SSE and direct request/response. Implemented by the SDK in `WebStandardStreamableHTTPServerTransport`.
- **Tool**: Client-invokable function in MCP. Each tool has a name, description, input schema (JSON Schema), and implementation.
